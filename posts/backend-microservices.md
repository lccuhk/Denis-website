# 我的第一个开源项目踩坑实录

> 开源不是银弹，它只是把你的问题从一个地方搬到了另一个地方，还多了一堆陌生人来围观。

## 先讲个故事

大二的时候，我脑子一热，想做一个开源项目。

"我要做一个牛逼的项目，拿到很多 Star，走上人生巅峰！"

然后我就踏上了开源的"不归路"。

几个月过去了，项目确实做出来了，Star 也有几十个（虽然大部分是同学和朋友点的），但坑也踩了不少。

今天就跟大家聊聊，我做第一个开源项目踩过的那些坑，希望你们别再踩了。

哦对了，我也是个学生，写的都是自己踩过的坑，不是什么大佬经验。

## 第一坑：项目规划稀烂

刚开始做项目的时候，我脑子一热就开始写代码。

"想那么多干嘛，先写再说！"

结果呢？

写了两个星期，发现方向不对，推翻重写。
又写了两个星期，发现架构有问题，又推翻重写。
前前后后推翻了三次，差点放弃。

**教训：**
- 动手写代码之前，先想清楚你要做什么
- 画个架构图，写个简单的设计文档
- 不用太详细，但至少要有个大概的方向
- 不然就是瞎忙活

我现在做项目，至少会花一两天时间规划，然后再动手写代码。虽然花了点时间，但总比写了一半推翻强。

## 第二坑：技术选型盲目追新

做项目的时候，我什么火就想用什么。

"微服务火，我要上微服务！"
"Kubernetes 火，我要上 K8s！"
"GraphQL 火，我要用 GraphQL！"

结果呢？

一个简单的个人项目，搞了十几个服务，部署起来麻烦得要死。
自己维护起来累得要死，还没什么用户。

**教训：**
- 技术是为业务服务的，不是为了炫技
- 简单的项目，用简单的技术栈就行
- 别为了用某个技术而用某个技术
- 适合的才是最好的

我后来把项目简化了，从十几个服务改成了一个单体应用，维护起来轻松多了。

## 第三坑：代码写得像屎

刚开始写代码的时候，我觉得能跑就行。

变量名随便起：`a`, `b`, `c`, `data`, `info`……
函数写得巨长，一个函数几百行。
注释？不存在的。我写的代码，我还看不懂吗？

结果呢？

过了一个月，我再看自己的代码，完全看不懂了。
"这写的啥玩意儿？"
"这个变量是干啥的？"
"这个函数是干啥的？"

更别说别人看我的代码了。

**教训：**
- 变量名要有意义，别用 `a`, `b`, `c`
- 函数别太长，一个函数最好不要超过 50 行
- 该写注释还是要写，特别是复杂的逻辑
- 代码是写给人看的，顺便给机器执行

我现在写代码，会特别注意这些。虽然慢一点，但至少一个月后我还能看懂。

## 第四坑：不写测试

"写测试？那不是浪费时间吗？"
"我自己手动测一下就行了。"

结果呢？

改了一个地方，三个地方出 Bug。
每次发布新版本，都要手动测半天，还不一定能测全。
用户提了 Bug，我还得花半天时间定位。

**教训：**
- 单元测试还是要写的，特别是核心逻辑
- 不用追求 100% 覆盖率，但关键路径一定要有测试
- 写测试看起来浪费时间，实际上能帮你省很多时间
- 有了测试，改代码也有底气

我现在写代码，会顺手写一些单元测试。虽然花了点时间，但改代码的时候放心多了。

## 第五坑：Git 用得稀烂

刚开始用 Git 的时候，我只会 `commit` 和 `push`。

"要那么多命令干啥，能提交就行。"

结果呢？

- 分支管理混乱，不知道哪个分支是干啥的
- Commit message 写得像鬼画符："更新"、"修复"、"优化"……
- 想回退到某个版本，不知道怎么操作
- 不小心把敏感信息提交上去了，不知道怎么删

**教训：**
- Git 是程序员的基本功，一定要学好
- 学会用分支，不同的功能用不同的分支
- Commit message 要写清楚，别人（包括未来的你）能看懂
- 学会用 `rebase`、`cherry-pick`、`stash` 这些高级命令
- 敏感信息（密码、密钥）绝对不能提交到 Git

我现在用 Git，会特别注意这些。虽然刚开始有点麻烦，但习惯了之后真的很方便。

## 第六坑：文档写得像鬼画符

"代码都写了，还写什么文档？"
"代码就是最好的文档！"

结果呢？

别人想用我的项目，不知道怎么用。
"这个怎么安装？"
"这个怎么配置？"
"这个 API 怎么调用？"

我得一个个解释，累得要死。

**教训：**
- README 一定要写清楚：项目介绍、怎么安装、怎么使用、怎么贡献
- API 文档要写清楚：每个接口的参数、返回值、示例
- 有能力的话，写个快速上手的教程
- 文档是给别人看的，也是给未来的自己看的

我现在的项目，README 写得特别详细。虽然花了点时间，但省了很多解释的功夫。

## 第七坑：不重视安全

"一个个人项目，哪有什么安全问题？"
"谁会攻击我啊？"

结果呢？

- 密码明文存在代码里，被人扫描到了
- SQL 注入漏洞，数据库被人拖库了
- XSS 漏洞，被人注入了恶意脚本

虽然是个人项目，没什么重要数据，但也够闹心的。

**教训：**
- 安全无小事，哪怕是个人项目
- 密码、密钥绝对不能硬编码在代码里
- 用户输入一定要校验，防止 SQL 注入、XSS 攻击
- 敏感数据要加密存储
- 定期更新依赖，修复安全漏洞

我现在写代码，会特别注意安全问题。虽然不能做到 100% 安全，但至少不会犯低级错误。

## 第八坑：不会处理 Issue 和 PR

项目开源后，有人提 Issue，有人提 PR。

我当时就慌了：

"这个 Issue 我不会修啊，怎么办？"
"这个 PR 写得不好，我要不要拒绝？怎么拒绝才不会得罪人？"

结果呢？

- Issue 堆了一堆，没人处理
- PR 要么直接合并（不管质量），要么直接拒绝（打击别人积极性）

**教训：**
- 别人提 Issue，不管能不能解决，先回复一下，表示你看到了
- 别人提 PR，先感谢，然后认真 review
- PR 有问题，友好地指出来，告诉别人怎么改
- 实在不想处理，可以加个 `help wanted` 标签，让别人来帮忙

我现在处理 Issue 和 PR，会特别注意态度。毕竟大家都是用爱发电，别伤了别人的心。

## 给想做开源项目的同学几个建议

### 1. 想清楚为什么要做开源
是为了学习？为了出名？还是为了找工作？
不同的目的，有不同的做法。
别盲目跟风，想清楚了再动手。

### 2. 从小项目开始
别一开始就想做一个"改变世界"的项目。
先从一个小工具、一个小库开始。
积累经验了，再做大项目。

### 3. 坚持很重要
开源项目不是一蹴而就的，需要长期维护。
可能做了几个月，也没几个用户，没几个 Star。
别灰心，坚持下去，总会有人看到的。

### 4. 学会接受批评
别人提 Issue，提 PR，是对你的项目感兴趣。
哪怕是批评，也是为了让项目变得更好。
别玻璃心，虚心接受，有则改之，无则加勉。

### 5. 别影响学业
学生嘛，学业还是第一位的。
别因为做开源项目，耽误了上课、考试。
得不偿失。

## 结语

做开源项目这一年，我学到了很多东西。

技术上的，非技术上的。

虽然踩了很多坑，但也成长了很多。

如果你也想做开源项目，别犹豫，动手去做。

踩坑不可怕，可怕的是不敢开始。

当然，该踩的坑，一个都少不了。

毕竟，踩坑才是最好的学习方式。

好了，不说了，还有个 Issue 没处理，还有个 Bug 没修。哦对了，明天还要交作业。

江湖路远，我们后会有期。
