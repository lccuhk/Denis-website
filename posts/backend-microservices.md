# 微服务踩坑实录

> 微服务不是银弹，它只是把你的问题从一个地方搬到了另一个地方。

## 先讲个故事

三年前，我接手了一个祖传单体项目。

代码乱得像一锅粥，改一个地方三个地方出 Bug。部署一次要半小时，回滚一次要一小时。

"我们要做微服务！"技术总监一拍桌子。

然后我们就踏上了微服务的"不归路"。

三年过去了，我想说——

微服务是真的香，但坑也是真的多。

今天就跟大家聊聊，我这几年踩过的那些坑。

## 第一坑：服务拆得稀碎

刚开始做微服务的时候，我们走了极端。

"一个表一个服务！"

"一个接口一个服务！"

结果呢？

一个简单的用户下单流程，要调用七八个服务。

用户服务 → 商品服务 → 库存服务 → 订单服务 → 支付服务 → 通知服务 → 物流服务

任何一个服务挂了，整个流程就挂了。

而且，服务多了，运维成本直接爆炸。

**教训：**
- 微服务不是拆得越细越好
- 先从业务边界开始拆，而不是从表开始
- 一个服务应该能独立完成一个完整的业务功能

## 第二坑：分布式事务

这个坑，谁踩谁知道。

单体的时候，一个事务就搞定的事，微服务下要了命。

用户下单，要扣库存，要创建订单，要扣余额。

任何一步失败了，怎么回滚？

我们试过：

### 方案一：TCC
Try-Confirm-Cancel。听起来很美，写起来想死。

每个接口要写三个逻辑，业务代码直接膨胀三倍。

而且，Confirm 和 Cancel 还要考虑幂等、重试……

### 方案二：本地消息表
这个相对靠谱一点。

把要做的事写到本地消息表，然后异步去执行。

但是，要自己写补偿逻辑，要处理各种异常情况。

### 方案三：Saga
长事务解决方案，但是实现复杂，回滚逻辑难写。

**教训：**
- 能不用分布式事务就不用
- 业务上尽量避免跨服务的强一致性需求
- 最终一致性才是微服务的常态

## 第三坑：服务间调用

服务拆了，就要互相调用。

这里的坑，比你想象的多。

### 坑一：超时设置
```java
// 错误示范
@FeignClient(name = "user-service")
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUser(@PathVariable("id") Long id);
}
```

默认超时时间是多少？10秒。

用户服务卡了，调用方也跟着卡。

一个慢服务，拖垮整个链路。

**正确姿势：**
```java
@FeignClient(name = "user-service", configuration = FeignConfig.class)
public interface UserClient {
    @GetMapping(value = "/users/{id}", timeout = 1000)
    User getUser(@PathVariable("id") Long id);
}
```

超时时间要根据业务场景设置，别太长也别太短。

### 坑二：重试机制
调用失败了，要不要重试？

要，但不能瞎重试。

```java
// 错误示范
@Retryable(maxAttempts = 10)
public User getUser(Long id) {
    return userClient.getUser(id);
}
```

用户服务挂了，你重试 10 次有什么用？

反而会把用户服务打挂。

**正确姿势：**
- 只对 GET 请求重试
- 只对 5xx 错误重试
- 重试次数别太多，3 次顶天了
- 要有退避策略

### 坑三：熔断降级
这个太重要了。

一个服务挂了，不能让它把整个系统拖垮。

用 Hystrix 或者 Sentinel，该熔断就熔断，该降级就降级。

```java
@SentinelResource(value = "getUser", fallback = "getUserFallback")
public User getUser(Long id) {
    return userClient.getUser(id);
}

public User getUserFallback(Long id, Throwable e) {
    // 返回默认值或者缓存值
    return new User();
}
```

## 第四坑：服务发现

服务多了，怎么知道每个服务在哪？

用 Eureka、Nacos、Consul……

但是，服务发现也有坑。

### 坑一：服务上下线不及时
服务挂了，注册中心还没感知到，请求还在往里面发。

结果就是一堆超时错误。

**解决：**
- 合理设置心跳时间
- 客户端要有健康检查
- 配合熔断降级

### 坑二：多环境问题
开发环境、测试环境、预发布环境、生产环境……

服务发现怎么隔离？

用不同的 namespace，或者不同的注册中心集群。

别图省事共用一个，迟早出大事。

## 第五坑：链路追踪

微服务下，一个请求走七八个服务。

出了问题，怎么排查？

靠日志？一个个去查？

别逗了，等你查到，用户都投诉完了。

一定要上链路追踪！

SkyWalking、Zipkin、Jaeger……随便选一个。

有了链路追踪，出了问题一眼就能看到：
- 哪个服务慢了
- 哪个服务报错了
- 整个调用链是什么样的

排查问题的效率，直接提升十倍。

## 第六坑：配置管理

服务多了，配置文件也多了。

改一个配置，要改十几个服务的配置文件？

别，用配置中心。

Nacos、Apollo、Spring Cloud Config……

统一管理配置，动态刷新，不用重启服务。

但是，配置中心也有坑：

### 坑一：配置优先级
本地配置、远程配置、环境变量……哪个生效？

一定要搞清楚优先级，不然怎么死的都不知道。

### 坑二：配置变更通知
配置改了，怎么通知到各个服务？

推模式还是拉模式？各有优劣。

### 坑三：配置版本管理
配置改坏了，怎么回滚？

一定要有版本管理，不然改坏了都不知道改了什么。

## 第七坑：日志管理

微服务下，日志散落在各个服务上。

查问题要登录十几台服务器去看日志？

别，用 ELK 或者 EFK。

把所有日志收集到一起，统一查询，统一分析。

但是，日志也有坑：

### 坑一：日志太多
把所有日志都收集起来？

别，存储成本会爆炸。

该打什么日志，不该打什么日志，要有规范。

### 坑二：日志格式不统一
每个服务的日志格式都不一样？

查询的时候想死。

一定要统一日志格式，最好用 JSON 格式。

### 坑三：没有 Trace ID
同一个请求的日志，怎么串起来？

一定要在入口生成 Trace ID，然后透传到各个服务。

这样查日志的时候，一个 Trace ID 就能把整个链路的日志都查出来。

## 第八坑：部署运维

微服务下，部署运维的复杂度直接翻倍。

十几个服务，每个服务要部署多实例，还要滚动更新、灰度发布……

不用 Docker 和 K8s，你会累死。

但是，Docker 和 K8s 本身就是一个大坑。

### 坑一：镜像太大
一个 JVM 应用，镜像做出来 1G 多？

用多阶段构建，用 Alpine 基础镜像，能小很多。

### 坑二：资源限制
不设置 CPU 和内存限制？

一个服务内存泄漏，把整个节点的内存吃光。

一定要设置资源限制。

### 坑三：健康检查
服务启动了，但还没初始化完成，请求就进来了？

一定要配置就绪检查和存活检查。

## 给想做微服务的朋友几个建议

### 1. 想清楚要不要做
不是所有项目都适合微服务。

如果你的团队就三五个人，业务也不复杂，老老实实做单体。

微服务的复杂度，需要足够的人力去扛。

### 2. 基础设施先行
别上来就拆服务。

先把基础设施搭好：
- 服务发现
- 配置中心
- 链路追踪
- 日志收集
- 监控告警
- 容器化部署

这些东西没有，微服务就是灾难。

### 3. 循序渐进
别一下子拆成十几个服务。

先拆一两个核心服务，跑通了，有经验了，再慢慢拆。

### 4. 制定规范
接口规范、日志规范、错误码规范、部署规范……

没有规范，每个人都按自己的来，最后就是一团糟。

### 5. 监控告警
这个太重要了。

没有监控，服务挂了你都不知道。

等用户来投诉，就晚了。

## 结语

微服务这东西，说难不难，说简单也不简单。

它能解决很多问题，但也会带来很多新问题。

关键是，你要想清楚，你是不是真的需要它。

如果你的单体还能扛，就别急着拆。

如果真的要拆，做好踩坑的准备。

毕竟，该踩的坑，一个都少不了。

好了，不说了，排查问题去了。
